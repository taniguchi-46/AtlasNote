# Code Review

Atlas Noteの変更セットを、正しさと利用者への実害から優先してレビューする。根拠のある欠陥、回帰、契約違反、重要なテスト不足だけをfindingとして報告し、コードは変更しない。

## 責務と境界を守る

レビュー対象のコード、差分、仕様、関連テストを読み、必要に応じて安全な検証コマンドを実行する。レビュー依頼をファイル編集、修正、stage、commit、push、PR作成、デプロイの許可として扱わない。

通常のコードレビューでは変更に関係するセキュリティ境界も確認する。セキュリティだけを目的とするPR、commit、branch、working tree、patch、指定パス、リポジトリのレビューや、脅威探索、攻撃経路分析、深い監査はセキュリティ専用Skillへ引き継ぐ。

レビューと修正を同時に依頼されても、このSkillの実行中はレビュー結果の提示で停止する。修正は、必要に応じて `$implementation-plan`、`$implementation`、`$refactoring` へ引き継ぐ。

## レビュー対象と比較基準を確定する

ユーザーが明示したPR、commit、branch、patch、ファイル、機能を最優先する。指定がない場合は現在のworking treeを対象とし、次を分けて確認する。

1. `git status --short`
2. `git diff -- <対象パス>` によるunstaged差分
3. `git diff --cached -- <対象パス>` によるstaged差分
4. 未追跡ファイルの内容

対象別に比較基準を決める。

- branch: 明示されたbaseとのmerge-baseからHEADまでを確認する。
- commit: 対象commitと親、または明示された範囲を確認する。merge commitではmainlineを推測しない。
- PRまたはpatch: 提供された差分とbase情報を使う。
- 指定ファイル: 指定された版または現在の内容と、正しさの判断に必要な履歴・周辺コードを確認する。

依頼なしにfetch、pull、checkout、branch作成を行わない。比較基準を一意に決められない場合は推測せず、確認できた範囲と必要なbaseを明記する。対象指定がなくworking treeも空の場合は、リポジトリ全体を勝手にレビューせず対象の指定を求める。

正式なfindingは、指定ファイル・機能の対象範囲、レビュー対象の変更、または変更によって新たに到達・悪化する問題に限定する。差分レビューでは変更と独立した既存問題を混ぜず、重大であれば「対象外の既存リスク」として分離する。

## 仕様とコードフローを調査する

最初に `README.md`、`docs/status.md`、`docs/rules/ai.md`、`docs/rules/architecture.md`、`docs/rules/conventions.md` を確認する。対象機能に応じて `docs/development/scopes/` と、「正とする」と明記された設計資料を選んで読む。

判断根拠は次の順で扱う。

1. ユーザーが明示した要求、完了条件、対象外
2. 対象機能の正本と明記された仕様
3. 公開API、型、DB schema、永続形式、構造化エラーなどの既存契約
4. 既存テストが保証する振る舞い
5. 変更前後の実装

ドキュメント、テスト、実装が矛盾し、正本を特定できない場合はfindingとして断定せず「確認事項」にする。テストが存在するだけで仕様適合とみなさず、テスト自体が誤った期待値を固定していないか確認する。

差分だけで結論を出さず、対象に関係する範囲で次を追跡する。

- 変更された関数、型、コンポーネントの全体
- 呼び出し元と呼び出し先
- 入力から出力、永続化、外部境界までのデータフロー
- 状態遷移、非同期順序、cancel、retry、shutdown
- 正常系、失敗系、部分成功、rollback、復旧
- 関連する既存テストと未検証の分岐

事実、コードから導ける推論、未確認事項を区別する。PowerShellで日本語ファイルを読む場合はUTF-8を明示する。`.env`、APIキー、パスワード、トークンなどの秘密情報は読まず、表示しない。

## 推奨順でレビューする

次の順で確認する。結果には該当する項目だけを記載し、全項目を形式的に列挙しない。

1. 正しさ・仕様適合
2. データ消失・破損
3. セキュリティ
4. 非同期・競合
5. エラーハンドリング
6. エッジケース・境界条件
7. 状態整合性
8. テスト
9. API・責務設計
10. 可読性・保守性
11. パフォーマンス
12. 命名・コメント
13. コードスタイル

確認順と最終Severityを分離する。正しさは最初に確認するが、限定的で復旧容易な仕様差までCriticalにしない。最終Severityは影響、到達可能性、発生可能性、復旧可能性を根拠に決める。

## Severityを判定する

### Critical: 最優先

最初に次をCritical候補として評価する。

1. 正しさ・仕様適合
2. データ消失・破損の可能性
3. セキュリティ
4. 重大な競合・非同期バグ

変更の中心目的が通常の利用経路でほぼ成立せず回避策もない、回復困難なユーザーデータ損失、現実的に悪用できる権限回避・秘密漏えい、上書き・deadlockなどの重大な競合が到達可能な場合にCriticalとする。限定的な機能欠落は影響に応じてHigh以下とし、mergeや公開を止める必要がある具体的な根拠を示す。

### High: 優先度高

次をHigh候補として評価する。

1. エラーハンドリング
2. エッジケース・境界条件
3. 状態管理の整合性
4. API・関数の契約
5. テスト不足

通常の利用経路で重要機能が壊れる、失敗を成功扱いする、状態またはAPI契約が不整合になる、重要な境界条件を処理できない場合にHighとする。Critical / Highの不変条件を変更したのに、その失敗を検出できるテストがない場合もHigh候補とする。

### Medium: 中優先度

次をMedium候補として評価する。

1. 可読性
2. 責務分離
3. 命名
4. 重複コード
5. 変更容易性
6. コメント

現在の差分に具体的な誤用、修正漏れ、将来変更時の回帰を招く構造上の問題がある場合にMediumとする。単なる好み、抽象的な「読みづらい」、将来の可能性だけではfindingにしない。

### Low: 低優先度

次をLow候補として評価する。

1. パフォーマンス
2. ログ・デバッグ容易性
3. コーディング規約
4. 細かな表現・スタイル

非ブロッキングだが具体的な改善価値がある場合だけLowとする。性能は測定値、計算量、hot path、現実的なデータ量の根拠を要求する。スタイルは変更行に対する明確なプロジェクト規約違反へ限定する。

領域名より実害を優先する。性能問題でもUI停止、DoS、保存失敗を起こすなら昇格し、正しさの問題でも影響が限定的なら適切なSeverityへ下げる。

## Atlas Noteの不変条件を確認する

対象変更に関係する行だけを確認する。

| 対象 | 確認する契約 |
| --- | --- |
| 保存・復旧 | Markdown本文を正本とし、操作ID付き一時ファイル作成 → SQLiteメタデータとoperation journalの同一transaction確定 → Markdown正本への置換 → journal削除の順序を守る。Markdown確定失敗時はSQLiteを戻し、補償失敗時は一時ファイルとjournalを復旧用に残す |
| revision・競合 | `expectedRevision` / CAS、stale要求の完全な無変更、一要求一revision、競合draft保持、自動merge・自動rebase・強制上書き禁止を守る |
| 非同期・状態 | ノート単位queue、単一in-flight、別lane分離、latest-wins、失敗lane停止、`flush` / `flushAll`、終了前flush、古い応答による選択逆転防止を守る |
| 不整合・復旧 | SQLiteレコードや孤児Markdownを自動削除せず、未完了操作と復旧情報を保持し、Wails API公開前に復旧する |
| DB・検索・タグ | SQL parameter binding、transaction、外部キー、UNIQUE、CASCADEを守る。FTS5を再構築可能な派生データとして扱い、索引失敗でMarkdown保存を戻さない。タグ操作でMarkdown、revision、`updated_at`、FTS5、保存journalを変更しない |
| セキュリティ・ログ | path traversalと危険なURLを防ぐ。raw HTMLはMarkdown正本に保持したまま、Rich変換時に実行可能なDOM、イベント属性、危険なURLとして解釈しない。本文、タイトル、検索語、ファイルパス、秘密情報、内部stackをログや利用者向けエラーへ出さない |
| API・責務 | 対象仕様で構造化結果と定めた競合などの既知エラーを文字列解析へ依存させず処理し、DB・I/O・内部障害は既存のGo error / Promise rejection契約を維持する。Component → Store / Composable / API client → Wails → Service → Repository / Storageの境界を守る |

## 根拠のあるfindingだけを報告する

各findingに次を含める。

- Severityと結果を表す短いタイトル
- 正確なファイルパスと、問題を理解できる最小の行範囲
- 問題が成立する入力、状態、実行順序
- 期待する挙動と実際の挙動
- 違反する仕様、契約、不変条件、またはコードフロー
- 利用者、データ、システムへの具体的な影響
- 最小限の修正方向
- 保証すべき回帰テスト

行番号は問題を生む式または変更行へ付ける。離れた複数箇所が原因でも主位置を狭く示し、本文で関連箇所を補足する。同じ根本原因から生じる複数の症状は一件へまとめる。Codexアプリでローカルファイルを示す場合は、絶対パスと行番号を含むクリック可能なMarkdownリンクを使い、Windowsでもリンク対象の区切りを `/` にする。

次をfindingにしない。

- 到達可能性を確認できない仮説
- 具体的なscheduleを示せない理論上のrace
- untrusted sourceからsinkまでの経路や欠けた防御を示せないセキュリティ懸念
- 実行していないテストが失敗すると断定した内容
- 変更していない既存コードの一般的なcoverage不足
- 明確な規約や実害に結び付かない命名・スタイルの好み
- 正本を特定できない仕様解釈

確度が低いが判断に重要な事項はfindingへ昇格させず、「確認事項」として必要な追加情報を示す。

## テスト不足を具体化する

「テストが少ない」だけではfindingにしない。次をすべて満たす場合にHighのテスト不足として報告する。

1. 差分がCriticalまたはHighの契約・不変条件へ触れている。
2. 正常系、失敗系、境界値、競合順序の重要な分岐を既存テストが保証していない。
3. その回帰が他の検証で検出されず混入し得る。

非重要領域の具体的なテスト不足は影響に応じてMedium以下とする。必要なテストでは、入力、発生条件、期待する出力・状態・順序を示し、内部関数の呼び出し回数だけを要求しない。機能上のfindingに同じ回帰テストが必要な場合は、独立したテスト不足として重複報告せず、そのfindingへ統合する。

## 安全に検証する

実行前に `package.json`、`frontend/package.json`、Go package、対象テストスクリプトを読み、書き込みや外部接続を伴わないことを確認する。対象に必要な最小コマンドだけを選ぶ。

安全性を確認できた場合は、必要に応じて次を使う。

- `git diff --check`
- 対象Go packageまたは `go test ./...`
- `npm run frontend:typecheck`
- `npm run frontend:lint`
- 関連する `npm --prefix frontend run test:<name>`

formatter、`--fix`、snapshot更新、code generation、migration、依存追加、アプリ起動を行わない。`npm run frontend:build` と `wails build` は成果物やbindingsを生成し得るため、レビューだけの依頼では実行しない。必要なら未実行の検証として提示する。

検証の前後で `git status --short` を比較する。コマンドが予期せずファイルを変更した場合は停止し、変更内容を報告する。ユーザーの変更や生成物を勝手に削除・復元しない。

テスト結果を成功、失敗、未実行に分ける。失敗はレビュー対象起因、既存不具合、環境要因、flakyの可能性を区別し、根拠なしに変更起因と断定しない。

## 出力形式

上位のユーザー指示または `AGENTS.md` に形式指定がある場合は優先する。指定された見出しの順序も維持する。次の形式では `### 結論` を件数だけにし、直後に `### 変更ファイル` を置き、`#### Findings` は必ず `### 変更内容` の内側へ置く。findingや概要を必須見出しより前へ移動しない。指定がない場合は、日本語のMarkdownで次を出力する。

```markdown
### 結論

Critical: 0 / High: 1 / Medium: 0 / Low: 0

### 変更ファイル

なし（レビューのみ）。

### 変更内容

#### Findings

##### [High] 保存失敗後にdirty draftが失われる

- 場所: `path/to/file.ts:123`
- 発生条件: 問題が起きる入力、状態、実行順序
- 期待する挙動: 仕様または契約が要求する状態
- 実際の挙動: コードフローから確認した状態
- 根拠: 関連仕様、契約、呼び出し経路
- 影響: 利用者、データ、システムへの実害
- 修正方向: 最小限の対応方針
- 必要なテスト: 保証すべき入力、出力、状態、順序

### 変更理由

レビュー対象、比較基準、参照した仕様を記載する。

### 確認方法

確認した差分と関連コード、実行したコマンド、成功・失敗・未実行を記載する。

### 注意点

確認事項、残余リスク、対象外の既存問題を記載する。ファイル編集、commit、pushを行っていないことも明記する。

### 次の行動

Critical、Highから優先し、必要な対応を1〜3件記載する。
```

findingはCritical → High → Medium → Lowの順、同一Severityでは影響が大きい順に並べる。findingがない場合は `#### Findings` の直下に「指摘事項なし。」だけを記載し、依頼がなければ肯定的な所感や変更概要を追加しない。この場合も「問題が絶対にない」「承認済み」とは断定せず、レビュー範囲は「変更理由」、未実行テスト、確認不能な仕様、残余リスクは「注意点」に記載する。件数を埋めるためにLow指摘を作らない。

## 制約

このSkillでは次を行わない。

- ソースコード、テスト、設定、ドキュメントの編集
- ファイルやディレクトリの作成、削除、リネーム
- formatter、自動修正、snapshot更新、code generation
- 依存関係の追加・更新
- DB schema変更とmigration
- stage、commit、push、branch操作、PR作成、デプロイ
- 実データまたは外部システムへの書き込み

レビュー結果を提示した時点で停止する。
