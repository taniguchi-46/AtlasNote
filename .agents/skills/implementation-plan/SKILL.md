# Implementation Plan

Atlas Noteの変更に着手する前に、既存実装とプロジェクトルールを調査し、実装担当者が主要な変更を追加調査なしで開始できる具体的な計画を作成する。

このSkillは `Analyze → Search → Plan` だけを担当する。実装、ファイル変更、Git操作は行わず、計画の提示後に停止する。

## 入力を整理する

次のいずれかを入力として扱う。

- 実装する機能
- 修正する不具合
- リファクタリング対象
- TODOに記載されたタスク
- 既存の設計案または改善要求

最初に次を整理する。

- 目的
- 解決する問題
- 完了条件
- 明示された制約
- 対象範囲
- 対象外

明示された完了条件を勝手に変更せず、依頼にない仕様を追加しない。軽微な不明点は仮定として明示し、計画を大きく左右する不明点だけを未確定事項として残す。

## プロジェクト情報を確認する

最初に次を読む。

1. `README.md`
2. `docs/status.md`
3. `docs/rules/ai.md`
4. `docs/rules/architecture.md`
5. `docs/rules/conventions.md`
6. `docs/development/scopes/scope.md`
7. 対象タスクに関連する開発ドキュメント、TODO、設計資料

必要に応じて次も確認する。

- `.agents/AGENTS.md`
- `package.json`、`frontend/package.json`、`go.mod`
- `.env.example`
- 既存ルーティング、Wails API、DBスキーマ、migration
- 認証・認可、入力検証、エラー処理、ログ処理
- 関連テストと実際のテストスクリプト
- Gitの作業ツリーと対象差分

`.env`、APIキー、パスワード、トークンなどの秘密情報は読まず、表示しない。

PowerShellで日本語を含むテキストを読む場合は、`Get-Content -Encoding utf8` のようにUTF-8を明示する。表示上の文字化けだけを理由にファイルを変更しない。

ドキュメントを調査の起点として使い、現在の実装状態は実コードでも確認する。情報が一致しない場合は次のように扱う。

- ユーザーが明示した要件と完了条件を最優先する。
- 「正とする」と明記された対象機能の仕様書を、意図する仕様の根拠にする。
- `docs/development/scopes/` を要求範囲、`docs/status.md` を実装状況の根拠として扱う。
- 実コードとテストを、現在の実際の振る舞いの根拠として扱う。
- ドキュメント同士、またはドキュメントとコードが矛盾し、正本を特定できない場合は、どちらかを黙って採用せず、相違点と計画への影響を「未確定事項」に記載する。

調査開始時と終了時に `git status --short` を確認し、このSkillによるファイル変更がないことを比較する。既存の変更はユーザーの作業として保持し、変更、削除、復元しない。

## 現在の実装を追跡する

対象機能に関連するコードを検索し、変更点だけでなくデータと制御の流れを追跡する。最低限、関連する範囲で次を確認する。

- エントリーポイント
- 主要な実装
- 呼び出し元と呼び出し先
- 状態管理
- 永続化処理
- 入力検証
- エラー処理とログ
- 非同期処理、競合、終了処理
- 関連テスト
- 類似実装と既存の命名・実装パターン

フロントエンドでは、Component、ComposableまたはAPIクライアント、Pinia、Wails Bridgeの責務境界を確認する。バックエンドでは、Wails API、Application Service、Repository、Squirrel、SQLite、Markdown Storageの責務境界を確認する。

Atlas Note固有の次の前提を維持する。

- Markdown本文を正本として扱う。
- SQLiteをメタデータ、関連、再構築可能な索引に使う。
- ローカルファーストを維持する。
- revision、`expectedRevision`、CAS、ノート単位の操作キュー、単一writer保証を壊さない。
- ComponentからWails APIを直接乱用せず、既存の責務境界を維持する。

確認できた事実、根拠のある推論、未確認事項を区別する。コードを部分的に読んだだけで計画を確定しない。

## 問題または実装差分を特定する

### バグ修正

次を可能性が高い順に整理する。

1. 現象
2. 原因候補
3. 最も怪しい点
4. 根拠
5. 原因を確認する方法
6. 根本原因と二次的な影響

症状を隠す一時対応と、根本原因を解消する恒久対応を区別する。根本原因を確認できない場合は、推測を事実として記載しない。

### 機能追加

次の流れで要求との差分を整理する。

```text
現在の状態
↓
不足している処理または責務
↓
必要な変更
↓
期待する状態
```

### リファクタリング

改善対象の具体的な問題、現在の責務、維持すべき振る舞い、変更が必要な根拠を整理する。要求と無関係な整理や将来のためだけの抽象化を計画に含めない。

## 変更範囲を決める

変更が必要なファイルと、影響確認だけが必要なファイルを区別する。各対象について次を明記する。

- ファイルパス
- 現在の責務
- 変更または確認する内容
- その理由

最小差分を優先し、無関係なリファクタリング、ライブラリ追加、設計変更、UI変更、ファイル移動を計画に含めない。必要な場合は、理由、影響範囲、リスク、代替案、最小変更案を示す。

DB変更が必要な場合は、必要性、既存データへの影響、主キー・外部キー・NULL・UNIQUE・INDEX・データ型、migration、rollback、本番適用時の注意点を計画する。migrationは実行しない。

## 影響とリスクを確認する

対象に関連する項目だけを選び、影響の有無と根拠を確認する。

- 既存機能と後方互換性
- Markdown本文、SQLite、ユーザーデータ
- Wails APIとフロントエンド状態
- 自動保存、未保存状態、保存失敗、終了前flush
- 非同期処理、順序逆転、競合、部分成功
- エラー処理、通知、ログ
- レスポンシブ対応、アクセシビリティ、ローディング、空・失敗状態
- 外部APIのタイムアウト、リトライ、レート制限、コスト

データ消失、上書き、競合、未保存状態に関係する変更は高リスクとして明示する。

セキュリティに関係する場合は、入力バリデーション、SQLインジェクション、XSS、CSRF、CORS、認証、認可、セッション、Cookie/JWT、ファイルパス、秘密情報、ログ、エラー情報、レート制限を確認する。関係しない項目を形式的に列挙せず、対象との関係を示す。

## テスト戦略を設計する

実装計画に必ずテスト戦略を含める。次を整理する。

- 追加するテスト
- 更新する既存テスト
- 維持する既存テスト
- 正常系、異常系、境界値
- 非同期処理の順序、競合、失敗、再試行
- 手動確認が必要な項目

バグ修正では、可能な限り問題を再現し、修正前に失敗して修正後に成功する回帰テストを計画する。内部関数の呼び出し回数ではなく、利用者から見た保証対象の振る舞いを基準にする。

確認コマンドはドキュメントの記載だけで決めず、実際の `package.json`、`frontend/package.json`、Goパッケージ、既存テストを確認して選ぶ。対象外の重いコマンドを機械的に追加しない。

調査中に実行できるのは、ファイル検索・読み取り、`git status`、`git diff` など、ワークスペースと外部状態を書き換えないコマンドだけとする。build、test、code generation、アプリ起動、依存関係のインストール、migrationは実行せず、確認コマンドとして計画に提示する。実行しなければ根本原因を確定できない場合は、推測で確定せず確認方法と未確定事項を記載する。

## 実装手順を作る

依存関係に沿って実装順序を決める。各ステップに必ず次を含める。

1. 変更対象
2. 変更内容
3. 変更理由
4. 関連する既存処理
5. テストまたは確認方法

「保存処理を改善する」のような抽象的な表現だけで終えない。変更する状態、条件、入出力、エラー時の挙動を具体化する。

## 出力形式

上位のユーザー指示または `AGENTS.md` に最終出力形式が指定されている場合は、その形式を優先し、次の項目を対応する見出しへ統合する。指定がない場合は、必ず次の形式で日本語のMarkdownを出力する。どちらの場合も、調査結果、関連ファイル、順序付き実装手順、テスト戦略、影響範囲、リスク、対象外、未確定事項を省略しない。

```markdown
# 実装計画

## 1. 結論

採用する実装方針を簡潔に記載する。

## 2. 調査結果

### 現在の実装

現在の処理、データフロー、確認した根拠を記載する。

### 問題または不足

根本原因、原因候補、または要求との差分を記載する。

### 関連ファイル

| ファイル | 現在の役割 | 対応 |
| --- | --- | --- |
| `path/to/file` | 現在の責務 | 変更 / 影響確認 |

## 3. 実装方針

採用する設計と理由を記載する。代替案を比較した場合は、不採用理由も簡潔に記載する。

## 4. 実装手順

### Step 1: タイトル

- 対象: `path/to/file`
- 変更内容: 具体的な変更内容
- 理由: 変更が必要な理由
- 関連する既存処理: 呼び出し元、呼び出し先、既存責務
- 確認: テストまたは確認方法

## 5. テスト戦略

### 追加・更新するテスト

保証する振る舞いを記載する。

### 維持する既存テスト

回帰させない振る舞いを記載する。

### 確認コマンド

実際のリポジトリ設定から確認したコマンドを記載する。

## 6. 影響範囲

影響する機能、データ、API、UIと、影響しない範囲を記載する。

## 7. リスクと注意点

対象変更に固有のリスク、軽減策、確認点を記載する。

## 8. 対象外

今回変更しない事項を記載する。

## 9. 未確定事項

調査しても判断できなかった事項だけを記載する。存在しない場合は「なし」と記載する。
```

上位の出力形式に「変更ファイル」がある場合は「なし（調査・計画のみ）」と記載する。ない場合は、計画の末尾に `ファイル変更: なし（調査・計画のみ）` と明記する。

## 制約

このSkillの実行中は、次の操作を禁止する。

- ソースコード、テストコード、設定、ドキュメントの編集
- ファイルやディレクトリの作成、削除、リネーム
- 依存関係の追加または更新
- DBスキーマ変更とmigration実行
- Git commit、push、ブランチの作成・切替・削除
- Pull Request作成、デプロイ、外部システムへの書き込み

ユーザーが実装も同時に依頼した場合でも、実装を開始しない。計画の末尾で、実装は別作業または実装用Skillへ引き継ぐ必要があることを伝えて停止する。
