# Refactoring

「コードを綺麗にすること」より先に「壊れにくくすること」を優先する。対象を限定し、維持する挙動をテストで固定してから、小さな単位で内部構造を改善する。

## 入力と適用範囲を確定する

最初に次を作業契約として確定する。

- 対象ファイル、モジュール、機能
- 改善する問題と完了条件
- 維持する外部挙動と不変条件
- 対象外
- ユーザーが指定した優先度または制約

明示された対象だけを調査・変更する。リポジトリ全体の一括整理、発見した対象外問題のついで修正、全体formatを行わない。

外部挙動、公開API、永続形式、DBスキーマ、構造化エラー契約、UI / UXを原則として維持する。確認済みの危険な挙動を正す必要がある場合は、単なる構造変更ではなく不具合修正を含むことを明示し、期待する挙動と回帰テストを確定してから変更する。

対象が広範囲、複数レイヤーの設計変更、仕様変更、migration、新規依存を伴う場合は、実装を開始せず `$implementation-plan` で計画を作成する。局所的で挙動維持の変更は、このSkill内で調査・実装・検証まで進める。

このSkillの呼び出しは、対象範囲内のローカルファイル編集とテスト実行の許可として扱う。stage、commit、push、PR、デプロイ、実データ変更、外部システムへの書き込みには拡張しない。

## 変更前の安全基準を作る

1. `git status --short` を確認し、対象についてworking tree差分、`git diff --cached -- <対象パス>` のstaged差分、未追跡ファイルの初期状態を分けて記録する。
2. `README.md`、`docs/status.md`、`docs/rules/ai.md`、`docs/rules/architecture.md`、`docs/rules/conventions.md`、関連仕様を確認する。
3. 対象の呼び出し元、呼び出し先、データフロー、状態遷移、非同期処理、失敗経路、永続化境界を追跡する。
4. 関連テストと実際のpackage scriptsを確認し、変更前の対象テストを実行する。
5. 維持する挙動、改善する内部構造、変更しない契約を明文化する。

PowerShellで日本語を含むファイルを読む場合はUTF-8を明示する。`.env`、APIキー、パスワード、トークンなどの秘密情報は読まず、表示せず、変更しない。

外部挙動または不変条件に触れる変更は、優先度を問わず既存テストで保証できるか確認する。保証できない場合は、挙動ベースのcharacterization testまたは回帰テストを変更前に追加する。High領域ではこの基準作成を必須とする。内部関数の呼び出し回数ではなく、入力、出力、状態、永続化、エラー、順序を固定する。

変更前から関連テストが失敗する場合は、変更起因と誤認しないよう結果を記録して原因を切り分ける。原因を特定できない重要な失敗がある状態ではリファクタリングを開始しない。

## High → Medium → Lowで優先する

依頼範囲内の根拠がある問題だけを分類する。優先度は評価と選択の順番であり、見つけた項目をすべて自動修正する権限ではない。各項目を「対応」「対象外」「保留」「該当なし」のいずれかにする。

分類名より失敗時の影響を優先する。境界条件や性能問題でもデータ損失、破損、保存失敗、競合、セキュリティ問題を起こす場合はHighとして扱う。命名やファイル移動でも広範囲の変更を伴う場合はLow相当として分離する。

### High: 最優先

1. データ損失・破損の防止
2. 非同期処理・競合状態の安全性
3. エラーハンドリング
4. 状態管理の整合性
5. 重要ロジックのテスト不足

今回変更するコードパスまたは維持契約に交差するHighを先にテストで固定して対応する。安全に対応できないHighが残る場合は部分完了として停止し、そのコードパスのMediumやLowへ進まない。同じファイルやモジュールで見つけても依頼目的と独立しているHighは対象外とし、勝手に修正せず注意点として報告する。

### Medium: 重要

1. 責務の分離
2. 重複コード
3. 関数・クラスの肥大化
4. モジュール間の依存関係
5. 型安全性
6. 命名
7. 境界条件

Highの安全基準を維持したまま改善する。行数だけで分割せず、責務と変更理由で境界を決める。同じコードに見えても不変条件が異なる処理は共通化しない。共通化にboolean分岐や巨大utilityが必要なら採用しない。`any`、型assertion、例外握りつぶしで見かけ上の整理を行わない。

### Low: 後回し

1. コードスタイルの統一
2. コメント改善
3. ファイル・ディレクトリ整理
4. パフォーマンス最適化
5. 過剰な抽象化の削除

ユーザーが明示した場合、またはHigh / Mediumの安全な変更に局所的に必要な場合だけ実施する。性能最適化は測定値と再現条件なしに行わない。ファイル移動や一括renameを別の改善へ混ぜない。

## 小さな単位で変更する

優先度が最も高い一つの問題を選び、次を繰り返す。

1. 変更前の不変条件と対象テストを確認する。
2. 一つの責務、重複、依存、型、境界だけを最小差分で変更する。
3. formatterと最小の対象テストを実行する。
4. 外部挙動と不変条件が変わっていないことを確認する。
5. 対象差分をレビューし、次の問題へ進む。

次を守る。

- 一つの差分へ複数の無関係な改善を混ぜない。
- 各diff hunkを、選択した問題、不変条件、またはテストのいずれかへ対応付ける。
- 既存の命名、型、エラー形式、責務境界を尊重する。
- ユーザーの既存変更を上書き、削除、復元しない。
- 生成元があるファイルを手編集しない。
- 不要コードを削除する前に、参照元、動的利用、生成処理、後方互換性を確認する。
- テストを通すためだけに期待値を変更せず、外部契約が変わった場合は変更を止める。
- 新規依存、将来用抽象化、公開範囲の拡大を行わない。

## Atlas NoteのHigh不変条件を守る

対象に関係する行だけを検証する。

| 対象 | 維持する不変条件 |
| --- | --- |
| 保存・復旧 | Markdown本文を正本とし、操作ID付き一時ファイル作成 → SQLiteメタデータとoperation journalの同一transaction確定 → 一時ファイルのMarkdown正本への置換 → journal削除、という順序を維持する。Markdown確定失敗時はSQLiteを戻し、補償失敗時は一時ファイルとjournalを復旧用に残す |
| revision・競合 | `expectedRevision` / CAS、stale更新の無変更、一要求一revision、競合draft保持、自動merge・自動rebase禁止を維持する |
| 非同期・状態 | ノート単位queue、単一in-flight、別lane分離、latest-wins、失敗lane停止、`flush`、終了前flush、選択応答逆転防止を維持する |
| エラー・ログ | I/O、DB、Wails、Promiseの失敗を握りつぶさず、本文、タイトル、検索語、ファイルパス、秘密情報をエラーやログへ出さない |
| DB・検索・タグ | transaction、schema version、rollback、FTS5の派生性、SQL parameter binding、外部キー、UNIQUE、CASCADE、タグ操作の非更新項目を維持する |
| セキュリティ・境界 | path traversal、raw HTML、危険なURLを防ぎ、Component → Store / Composable / API → Wails → Service → Repositoryの境界を壊さない |

セキュリティはHigh / Medium / Lowとは別の必須ゲートとして扱い、入力検証、SQL、XSS、ファイルパス、認証・認可、秘密情報、ログを対象との関係に応じて確認する。これらの不変条件を変更する必要が生じた場合はリファクタリングの範囲を超えるため停止する。

## テストを実行する

ドキュメントの固定一覧ではなく、現在の `package.json`、`frontend/package.json`、Goパッケージ、既存テストからコマンドを選ぶ。

Highでは次の観点を優先する。

- データ損失・破損: 失敗、rollback、補償、復旧、再実行、冪等性
- 非同期・競合: 重複、順序逆転、同時実行、失敗、再試行、flush、終了
- エラー処理: DB、ファイルI/O、Wails、Promise失敗の伝播と秘密情報非出力
- 状態整合性: 正常遷移、失敗遷移、競合、切替、再読込、stale応答
- 重要ロジック: 正常系、異常系、境界値、回帰ケース

純粋なリファクタリングのcharacterization testは変更前後とも成功させる。変更前に失敗し変更後に成功するテストは不具合修正を表すため、修正を含むことを変更内容へ明記する。非同期テストは制御可能な順序、barrier、fakeを使い、固定時間のsleepだけに依存しない。

対象テストから始め、最後に変更リスクに応じて広い確認を行う。必要に応じて、変更したGoファイルへ `gofmt`、対象Go package、`go test ./...`、関連する `npm --prefix frontend run test:<name>`、`npm run frontend:typecheck`、`npm run frontend:lint`、`npm run frontend:build` を選ぶ。Wails境界へ影響する場合だけ `wails build` を実行する。

テスト失敗は変更起因、既存不具合、環境要因、flakyの可能性に分類する。変更起因なら範囲内で修正して再実行する。既存不具合や環境要因を成功扱いせず、依頼外の修正へ広げない。

## 最終差分をレビューする

1. `git diff --check`
2. `git status --short`
3. 変更済み・未追跡ファイルを含む対象差分
4. 変更前後の対象テスト結果
5. High / Medium / Lowの対応・保留状況
6. 計画外ファイル、不要生成物、デバッグコード、秘密情報、外部挙動の変化がないこと

範囲内の問題を修正した後、必要なテストを再実行する。`git reset --hard`、ユーザー変更のrevert、stage、commit、push、ブランチ操作は行わない。

## 停止条件

次の場合は実装を停止し、根拠と必要な判断を報告する。

- 対象、維持する挙動、完了条件を確定できない。
- 外部挙動または不変条件に触れる変更を安全に固定するテストを追加・実行できない。
- 変更前から重要テストが失敗し、原因を切り分けられない。
- ユーザーの既存変更と競合し、安全に保持できない。
- 公開API、DBスキーマ、永続形式、UI / UX、認証境界、構造化エラー契約の変更が必要になる。
- 新規依存、大規模ファイル移動、複数レイヤーの設計変更が必要になる。
- 実DB、ユーザーデータ、外部システムへの書き込み、破壊的操作、追加権限が必要になる。
- 性能最適化に測定値と再現条件がない。
- ドキュメントと実装の矛盾が結果を左右し、正本を特定できない。

軽微な内部実装の違い、局所的なファイル位置、既存パターンで解決できる命名だけでは停止しない。

## 完了条件

- 明示された完了条件を満たした。
- 外部挙動と対象の不変条件を維持した。
- 対象内に未対応のHighを残していない。保留する場合はリファクタリングを部分完了として報告した。
- 目的とした構造上の問題を減らし、新しい不要な抽象化や依存を増やしていない。
- 変更起因の必須テストと関連回帰テストが成功した。
- 最終差分が対象範囲内で、ユーザーの既存変更を保持している。

## 出力形式

上位のユーザー指示または `AGENTS.md` に形式指定がある場合は優先する。指定がない場合は、日本語のMarkdownで次を出力する。

```markdown
### 結論

完了、部分完了、未完了を明記する。

### 変更ファイル

変更したファイルを列挙する。

### 変更内容

High / Medium / Lowごとに、発見事項、対応、保留、対象外を整理する。

### 変更理由

維持した不変条件と、内部構造を改善した理由を説明する。

### 確認方法

変更前後の結果を分け、実行したコマンドと成功・失敗・未実行を記載する。

### 注意点

対象外で見つけたHigh、保留事項、既存不具合、環境要因、未確認事項を記載する。commitとpushを行っていないことも明記する。

### 次の行動

必要なものだけを1〜3件記載する。
```
