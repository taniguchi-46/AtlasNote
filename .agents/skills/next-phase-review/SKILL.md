# Next Phase Review

現在フェーズの未完了、欠陥、技術的負債が次フェーズの実装を壊す、妨げる、または大きな手戻りを生むかを評価する。コードが動作することだけでなく、現在フェーズの出口条件と次フェーズの入口条件を満たすことを確認する。

## 役割と境界を守る

レビュー対象のコード、仕様、テスト、ドキュメントを調査し、必要に応じて安全な検証コマンドを実行する。ソースコード、テスト、設定、ドキュメントは編集しない。formatter、自動修正、snapshot更新、code generation、migration、stage、commit、push、ブランチ操作、PR作成、デプロイ、実データまたは外部システムへの書き込みを行わない。

修正、実装計画、実装を同時に依頼されても、このSkillでは移行判定と根拠の提示で停止する。CriticalまたはHighの対応が必要なら、必要に応じて $implementation-plan、$implementation、$refactoring へ引き継ぎ、対応後にこのSkillを再実行する。

## 対象と判断基準を確定する

最初に、現在フェーズ、次フェーズ、対象ブランチまたは作業ツリー、明示された出口条件と入口条件を確認する。ユーザーが指定していない場合は、ロードマップ、現在状況、TODO、詳細スコープから合理的に特定し、その仮定を報告する。現在フェーズまたは次フェーズを特定できず、入口条件も確定できない場合は、移行を承認せず NO-GO（判定前提不足）とする。

次の順で根拠を扱う。

1. ユーザーが明示した現在・次フェーズ、完了条件、制約
2. 対象機能で「正とする」と明記された仕様・設計資料
3. docs/development/scopes/、TODO、docs/status.md のフェーズ定義と進捗
4. API、型、DB schema、永続形式、構造化エラーなどの既存契約
5. 実装と関連テストが示す実際の振る舞い

開始時に git status --short を確認し、未コミット変更を現行フェーズの未反映作業か、ユーザーの別作業かに分けて記録する。ユーザーの変更を編集、削除、復元しない。終了時にも同コマンドを実行し、このSkillによる変更がないことを確認する。

少なくとも次を読む。

1. README.md
2. docs/status.md
3. docs/rules/ai.md
4. docs/rules/architecture.md
5. docs/rules/conventions.md
6. docs/development/scopes/scope.md と現在・次フェーズの詳細スコープ
7. 現在フェーズのTODO、実装計画、対象機能の設計資料
8. 関連する実装、呼び出し元・先、API、DB schema・migration、認証・入力検証、ログ、テスト
9. package.json、frontend/package.json、go.mod、.env.example のうち確認に必要なもの

.env、APIキー、パスワード、トークンなどの秘密情報は読まず、表示しない。PowerShellで日本語を含むファイルを読む場合はUTF-8を明示する。

## 出口条件と入口条件を対応付ける

現在フェーズの各出口条件について、仕様上の根拠、実装・テスト上の根拠、未確認部分を整理する。次に、次フェーズの各機能を開始するための既存機能、データ、API、責務境界、テスト基盤を逆算する。

次フェーズの機能 → 必要な既存機能・不変条件・データ → 現在の実装・テスト・ドキュメント上の根拠 → 欠落、矛盾、移行リスク

現在フェーズの実装が存在しても、次フェーズの前提を満たさない場合は、その前提不足をfindingとして扱う。ドキュメントの記述だけ、テストの存在だけ、または正常系の目視だけで出口条件を満たしたと判断しない。

## 優先順にレビューする

該当する項目だけを報告し、根拠のない懸念や好みをfindingにしない。領域名ではなく、利用者・データ・次フェーズへの実害を優先してSeverityを決める。

### Critical: 移行を停止する事項

| 観点 | 確認すること |
| --- | --- |
| データ消失・データ破損 | Markdown正本、SQLiteメタデータ、操作journal、migration、削除、補償、復旧の順序が、失敗時も既存データを失わないことを確認する。 |
| Correctness・仕様違反 | 現在フェーズの主要ユースケース、異常系、境界値、完了条件が仕様どおりかを確認する。 |
| 状態整合性・競合・非同期処理 | stale state、応答逆転、race、二重実行、複数writer、debounce中の消失、保存キュー、CAS、終了前flushを確認する。 |
| セキュリティ上の重大問題 | path traversal、任意ファイルアクセス、SQL injection、XSS、コマンド注入、権限境界、秘密情報・本文の露出を確認する。ローカルアプリでも入力経路を確認する。 |
| 回復不能なエラー | retry、rollback、復旧、dirty stateの保持、UIと内部状態の再同期を確認する。アプリ再起動以外に回復できない状態を見逃さない。 |

### High: 移行前に原則対応する事項

| 観点 | 確認すること |
| --- | --- |
| 現在フェーズの未完了・部分実装 | TODO、FIXME、stub、mock、暫定fallback、未使用の旧実装、完了扱いとの矛盾を確認する。 |
| アーキテクチャ上の拡張阻害 | 次フェーズの変更点に対し、UI・ドメイン・Service・Repository・Storageの境界、データモデル、API、依存方向、循環依存を確認する。 |
| 技術的負債による手戻りリスク | 次フェーズで変更することが確実なホットスポットの責務過多、密結合、重複、暗黙依存を確認する。単なる見た目の不統一は含めない。 |
| エラーハンドリング不足 | 失敗を握り潰さず、UIまたは呼び出し元へ適切に伝播し、再試行・部分成功・rollback・原因追跡が可能かを確認する。 |
| テスト不足 | データ保全、状態整合性、競合、失敗、境界値、主要ユースケースのうち、次フェーズで回帰させると重大な経路が検証されているかを確認する。 |

### Medium: 開始前に改善を検討する事項

| 観点 | 確認すること |
| --- | --- |
| API・責務境界 | 巨大なAPI、呼び出し順序への暗黙依存、UIからのデータ層直接操作、ServiceとRepositoryの責務重複を確認する。 |
| 保守性 | 複雑な条件分岐、巨大関数・コンポーネント、深いネスト、重複、magic value、分かりにくい副作用を確認する。 |
| ディレクトリ・モジュール構成 | feature境界、sharedやutilsへの過剰集約、UI・domain・infrastructureの混在、循環importを確認する。 |
| 依存関係 | 不要・重複・非推奨の依存、将来機能と競合する依存、固定理由が不明な重要バージョンを確認する。 |
| ドキュメントと実装の乖離 | status、TODO、スコープ、実装計画、技術資料と、コード・テストの実際の状態を照合する。 |
| 次フェーズ前提条件 | 次フェーズの設計未確定、必要なデータやAPIの欠落、入口条件の未検証を確認する。 |

### Low: 移行を妨げない改善事項

| 観点 | 確認すること |
| --- | --- |
| 命名 | 意味の曖昧さ、責務との不一致、略語過多、規則不統一を確認する。 |
| コメント | 古いコメント、コードの翻訳だけのコメント、複雑な処理のWhy不足を確認する。 |
| コードスタイル | formatterやlinterで扱える書式、import順、軽微な統一を確認する。 |
| 軽微なパフォーマンス改善 | 測定値、計算量、hot path、次フェーズで増えるデータ量の根拠がない微小最適化を確認する。データ量増加が確定している場合はMedium以上へ再評価する。 |

## Atlas Noteの高リスク経路を追跡する

対象に関係する場合、Component → Store / Composable / API client → Wails → Service → Repository / Storageの境界を追跡する。次の契約を特に確認する。

- Markdown本文を正本とし、SQLiteはメタデータ・関連・再構築可能な索引として扱うこと。
- Markdown更新、SQLite transaction、operation journal、補償、起動復旧の順序が失敗時も整合すること。
- expectedRevision / CAS、競合draft保持、ノート単位操作lane、単一writer、latest-wins、失敗lane停止、flush / flushAll、終了前flushを守ること。
- FTS5、タグ、リンクなどの派生索引の失敗がMarkdown正本の保存を壊さず、再構築できること。
- SQLのパラメータ化、入力検証、許可リスト、危険なURL・raw HTMLの安全な扱い、ログ・利用者向けエラーでの本文・秘密情報・内部情報の非露出を守ること。
- 既知の構造化エラーを文字列解析に依存せず、ComponentがWails APIやDB層を直接乱用しないこと。

## 根拠のあるfindingだけを報告する

各findingに必ず次を含める。

- Priority: Critical / High / Medium / Low
- 問題: 短いタイトルと、成立する入力、状態、または操作順序
- 根拠: 正確なファイルパスと最小の行範囲、仕様・テスト・コマンド結果、呼び出し経路
- 影響: 現在フェーズの利用者、データ、システムへの実害
- 次フェーズへの影響: 阻害される機能、増幅するリスク、見込まれる手戻り
- 推奨対応: 既存設計を維持する最小の対応方針と、必要な回帰テスト
- 移行前の対応: 必須 / 条件付き / 不要

行番号は問題を生む式または契約違反の位置へ付ける。Codexアプリでファイルを示す場合は、絶対パスと行番号を含むクリック可能なMarkdownリンクを使う。同じ根本原因から生じる症状は一件にまとめる。

次をfindingにしない。

- 到達可能性、入力、状態遷移を示せない仮説
- 具体的な実行順序を示せない理論上のrace
- untrusted source、sink、欠けた防御を示せないセキュリティ懸念
- 実行していないテストが失敗すると断定する内容
- 一般論だけのcoverage不足、命名、スタイル、性能改善
- 正本を特定できない仕様解釈

不確実だが移行判断に必要な事項はfindingへ昇格させず、「確認事項」として事実、未確認理由、判断に必要な追加情報を分けて記載する。ただし、データ保全・セキュリティ・次フェーズの必須前提を確認できない場合は、確認不能であること自体を移行判断へ反映する。

## 移行判定を行う

各Severityの件数だけで機械的に判定せず、現在フェーズの出口条件、次フェーズの入口条件、復旧可能性、対応の期限を合わせて判断する。

| 判定 | 条件 |
| --- | --- |
| GO | Criticalがなく、次フェーズを阻害するHighがなく、現在フェーズの出口条件と次フェーズの必須前提を根拠付きで確認できる。 |
| CONDITIONAL GO | Criticalがなく、未解決Highが次フェーズの開始・安全性・必須前提を阻害しない。各Highに影響範囲、暫定軽減策、対応方針、依存する作業より前の期限が明確にある。Highは原則として移行前に修正する。 |
| NO-GO | Criticalがある、現在フェーズの主要な出口条件が未達または検証不能、次フェーズの必須前提が欠ける、または未解決Highが次フェーズで増幅・再作業を招く。フェーズや入口条件を特定できない場合も NO-GO（判定前提不足）とする。 |

findingがないことだけをGOの根拠にしない。レビュー範囲、未実行の検証、確認不能な仕様、残余リスクを結論へ反映する。

## 安全に検証する

実行前に対象スクリプトと影響を確認し、ワークスペースや外部状態を書き換えない最小のコマンドだけを選ぶ。安全性を確認できた場合は、必要に応じて git diff --check、対象Go packageまたは go test ./...、npm run frontend:typecheck、npm run frontend:lint、関連する npm --prefix frontend run test:<name> を実行する。

npm run frontend:build、wails build、アプリ起動、formatter、--fix、snapshot更新、code generation、migration、依存関係の追加・更新は実行しない。検証の前後で git status --short を比較し、予期しない変更が生じた場合は停止して報告する。

結果は成功、失敗、未実行に分ける。失敗はレビュー対象起因、既存不具合、環境要因、flakyの可能性を根拠付きで区別する。

## 出力形式

上位のユーザー指示または AGENTS.md の形式を優先する。指定がない場合は、日本語のMarkdownで次を出力する。

### 結論

判定: GO / CONDITIONAL GO / NO-GO

Critical: 0 / High: 1 / Medium: 0 / Low: 0

判定理由を、現在フェーズの出口条件と次フェーズの入口条件に結び付けて簡潔に記載する。

### 変更ファイル

なし（レビューのみ）。

### 変更内容

#### Findings

##### [High] 次フェーズの保存経路が未確定

- Priority: High
- 場所: path/to/file:123、または仕様の該当節
- 問題: 成立する入力、状態、操作順序
- 根拠: コード、仕様、テスト、コマンド結果
- 影響: 現在フェーズへの具体的な実害
- 次フェーズへの影響: 阻害される機能または手戻り
- 推奨対応: 最小の対応方針と必要な回帰テスト
- 移行前の対応: 必須 / 条件付き / 不要

### 変更理由

現在フェーズ、次フェーズ、レビュー対象、参照した正本、出口条件と入口条件の対応を記載する。

### 確認方法

確認した実装・テスト・ドキュメントと、実行したコマンドを成功・失敗・未実行に分けて記載する。

### 注意点

確認事項、未確認事項、残余リスク、対象外の既存問題を記載する。ファイル編集、commit、pushを行っていないことを明記する。

### 次の行動

Critical、Highから優先し、必要な対応を1〜3件記載する。

findingはCritical → High → Medium → Lowの順に並べる。findingがない場合は Findings の直下に「指摘事項なし。」と記載する。肯定的な所感だけで移行を承認せず、未確認の出口条件・入口条件は「注意点」へ残す。
