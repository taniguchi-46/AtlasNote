# Implementation

承認済みの実装計画を作業契約として扱い、`Implement → Test → Review → Complete` を実行する。計画を最初から作り直さず、既存設計を維持した最小差分で完了条件を満たす。

`$implementation-plan` の末尾にある「変更ファイル: なし」は計画作成時に編集しなかったという意味であり、このSkillの実装禁止を意味しない。

## 入力契約を確認する

入力された計画から、見出し名に依存せず次を読み取る。

- 目的と完了条件
- 現在の実装と解決する問題
- 変更対象と影響確認対象
- 順序付きの実装手順
- テスト戦略と確認コマンド
- 影響範囲とリスク
- 対象外と未確定事項
- ユーザーが追加した制約

少なくとも目的、完了条件、対象範囲、対象外、順序付き手順を実装契約として確定する。低リスクの小変更では、テスト戦略と影響範囲を既存コードやscriptsから補完できる。DB、公開API、認証、データ整合性、削除、UI挙動に関わる場合は、テスト戦略とリスクも入力で確認できなければ実装しない。

計画を指定した実装依頼は、計画範囲内のローカルファイル編集とテスト実行の許可として扱う。commit、push、PR、デプロイ、既存または実データへのmigration適用、外部システムへの書き込み、秘密情報へのアクセス許可には拡張しない。承認済み計画にmigration実装が含まれる場合は、migrationファイルの編集と、テスト専用の破棄可能な一時DBに対する検証だけを行える。

計画がない場合、またはAPI、DB、データ整合性、認証、UI挙動、削除処理を左右する未確定事項が残る場合は、実装を開始せず `$implementation-plan` での計画策定または更新を案内する。軽微な実装詳細は既存パターンから判断し、毎回ユーザーへ確認しない。

## 計画と現在の状態を照合する

実装前に短い事前確認だけを行う。

1. `git status --short` と対象差分を確認し、既存のユーザー変更を記録する。
2. `README.md`、`docs/status.md`、`docs/rules/ai.md`、`docs/rules/architecture.md`、`docs/rules/conventions.md`、対象機能の仕様書を確認する。
3. 計画対象のコード、呼び出し元、呼び出し先、関連テスト、実際のpackage scriptsを確認する。
4. 計画の目的、完了条件、対象外、実装順序を実行チェックリストにする。
5. 計画した回帰テストで高リスク変更を検証できるか確認する。

PowerShellで日本語を含むファイルを読む場合はUTF-8を明示する。`.env`、APIキー、パスワード、トークンなどの秘密情報は読まず、表示せず、変更しない。`.env.example` は設定項目の確認に限って読める。

計画とコードのファイル名や行位置が軽微にずれていても、目的、責務、完了条件が同じなら現在の構造へ合わせて実装する。公開契約、データ構造、責務境界、ユーザー挙動が変わるずれは計画を黙って書き換えず、停止条件として扱う。

## 計画順に実装する

各ステップを次のループで処理する。

1. 対象ファイルと既存責務を再確認する。
2. 既存の命名、型、エラー形式、責務境界に合わせて最小差分で編集する。
3. 計画されたテストを追加または更新する。
4. formatterと最小の対象テストを実行する。
5. 失敗を切り分け、変更起因で範囲内なら修正して再実行する。
6. 差分がそのステップの目的だけに収まることを確認して次へ進む。

次を守る。

- 無関係なリファクタリング、将来用の抽象化、ファイル移動、命名変更を行わない。
- 計画にない依存関係、DB変更、公開API変更、UI / UX変更を追加しない。
- ユーザーの既存変更を上書き、削除、復元しない。重なる場合は内容を理解してから統合する。
- 問題を隠す例外握りつぶし、検証回避、型安全性の低下を採用しない。
- 生成元が存在するファイルを手編集せず、既存の生成手順を使う。不要な生成物を変更対象へ含めない。
- 仕様またはプロジェクトルール上必要な場合だけ、関連ドキュメントやタスク管理ファイルを更新する。

バグ修正では可能な限り再現テストを先に追加し、症状ではなく根本原因を修正する。既存テストだけで再現できる場合は、不要なテストを重複追加しない。

## 対象別の検証ゲート

関連する行だけを適用する。

| 対象 | 必須確認 |
| --- | --- |
| 保存・並行処理 | Markdown正本、SQLiteメタデータ、操作journal、revision / CAS、ノート単位queue、単一writer、保存失敗、終了前flushを壊さない |
| DB・migration | migrationファイルの編集は承認済み計画内に限定し、テスト専用の破棄可能な一時DBで、既存データ、制約、INDEX、schema version、rollback、失敗時の不変性を検証する。既存DB、ユーザーデータ、共有環境、本番へ適用しない |
| API・入力 | 入力検証、型、エラー契約、認証・認可、Repository境界、SQLのパラメータ化を維持する |
| フロントエンド | ローディング、空、失敗、非同期応答順、アクセシビリティ、既存デザインを確認する。ComponentからWails APIを直接乱用しない |
| エディタ・外部データ | raw HTML、危険なURL、XSS、path traversal、ファイル名へのユーザー入力利用を確認する |
| AI・外部API | APIキーを環境管理し、タイムアウト、リトライ、レート制限、コスト、途中切断、ログへの本文・秘密情報出力を確認する |

高リスク領域の回帰テストが計画に不足していても、既存のテスト構造と完了条件から範囲内で安全に補完できる場合は追加して継続する。新規依存、設計変更、未確定仕様、重大な範囲拡張が必要な場合だけ、実装を進めず計画の更新を求める。

## テストを実行する

ドキュメントの固定一覧ではなく、現在の `package.json`、`frontend/package.json`、Goパッケージ、既存テストからコマンドを選ぶ。

次の順で、変更に関係する範囲だけを実行する。

1. 変更ファイルのformatterまたは静的確認
2. 追加・更新した回帰テスト
3. 直接影響する既存テスト
4. リスクに応じた型チェック、build、広い回帰テスト

Atlas Noteでは必要に応じて次を選ぶ。

- 変更したGoファイルへ `gofmt`
- 対象Go packageのテスト後に `go test ./...`
- 関連する `npm --prefix frontend run test:<name>`
- `npm run frontend:typecheck`、`npm run frontend:lint`、`npm run frontend:build`
- Wails連携またはbindingsへの影響がある場合だけ `wails build`

`frontend/wailsjs/` や `frontend/dist/` などの生成物を手編集しない。すべてのコマンドを機械的に実行せず、重複する検証は変更リスクに応じて選ぶ。

テスト失敗を無視しない。次のように分類する。

- 変更起因かつ範囲内: 修正して、失敗したテストと関連回帰テストを再実行する。
- 既存不具合: 根拠を示し、依頼外の修正を行わない。
- 環境要因: 原因と再実行方法を記載し、成功扱いにしない。
- flakyの疑い: 根拠を確認して必要な再実行だけ行い、無限に再試行しない。

重要な検証を実行できない場合は「実装完了」と「検証未完了」を区別する。

## 最終差分をレビューする

実装後に次を確認する。

1. `git diff --check`
2. `git status --short`
3. 変更済み・未追跡ファイルを含む対象差分
4. 計画外ファイル、不要な生成物、デバッグコード、秘密情報、本文やpayloadのログ出力
5. ユーザーの既存変更が保持されていること
6. 完了条件とテスト結果が一致すること

差分レビューで見つけた範囲内の問題は修正し、必要なテストを再実行する。`git reset --hard`、ユーザー変更のrevert、stage、commit、push、ブランチ操作は行わない。

## 停止条件

次の場合だけ実装を停止し、根拠と必要な判断を報告する。

- 計画がなく、完了条件または対象範囲を確定できない。
- 現在のコードが計画前提から大きく変わり、計画どおりでは完了条件を満たせない。
- ユーザーの既存変更と競合し、安全に保持できない。
- 計画外の依存追加、DBスキーマ変更、公開API変更、UI / UX変更、認証境界変更、削除、大規模リファクタリングが必要になる。
- 既存DB、ユーザーデータ、共有環境、本番へのmigration適用、破壊的操作、外部システムへの書き込み、追加の権限が必要になる。
- 重要テストの修正が依頼範囲を超える、または高リスク変更を安全に検証できない。

軽微な命名、ファイル位置、内部実装の違いだけでは停止せず、既存パターンに合わせて吸収し、最終報告へ記載する。

## 完了条件

次をすべて満たした場合だけ完了とする。

- 計画の実装ステップと明示された完了条件を満たした。
- 変更起因の必須テストが成功した。
- 最終差分が計画範囲内で、既存機能とユーザー変更を壊していない。
- 未解決のセキュリティ、データ損失、競合リスクがない。
- 未確認事項と実行できなかった検証を正確に区別した。

## 出力形式

上位のユーザー指示または `AGENTS.md` に形式指定がある場合は優先する。指定がない場合は、日本語のMarkdownで次を出力する。

```markdown
### 結論

完了、部分完了、未完了を明記する。

### 変更ファイル

変更したファイルを列挙する。

### 変更内容

計画の各ステップに対応する変更を説明する。

### 変更理由

完了条件との関係と、必要だった軽微な計画調整を説明する。

### 確認方法

成功、失敗、未実行を分け、実行したコマンドと結果を記載する。

### 注意点

既存不具合、環境要因、未確認事項、影響範囲を記載する。commitとpushを行っていないことも明記する。

### 次の行動

必要なものだけを1〜3件記載する。
```

バグ修正では「変更理由」に現象、原因候補、最も怪しい点、根拠、根本原因、一時対応と恒久対応を必要な範囲で含める。
