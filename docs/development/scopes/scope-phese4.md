## 目的

ユーザー自身が選択したAIプロバイダーを利用し、ローカルファーストのノートを要約・整理・検索・執筆支援に活用できるようにする。AIを利用できない場合も、既存のローカル保存・編集・検索・同期を継続できることを前提とする。

## 対象範囲

### AI設定

- API Keyまたはローカルプロバイダーの認証設定
- プロバイダー切り替え
- モデル選択、モデルごとの能力・上限表示
- 接続確認、認証失敗、利用不能、レート制限、タイムアウトの表示
- Gemini、Groq、OpenRouter、OpenAI、Ollama、LM Studioを候補とする。ただし、各プロバイダーの対応可否と契約は実装前に確定する。

### AI司書

- メモ要約
- タイトル生成
- タグ生成
- 自動分類
- 関連メモ提案
- 重複メモ検出

### AIアシスタント／ライティング

- メモQ&A、RAG検索、アイデア壁打ち、ブレインストーミング
- プロンプト生成・改善
- README、ドキュメント、ブログ記事、要件定義の生成

## 実装前に確定する境界

### 1. AI認証・秘密情報

- API Key、アクセストークン、ローカル接続情報の保存先とライフサイクルを決める。
- OSのCredentialStore、セッション限定保持、未設定状態の優先順位を決め、平文のSQLite、Markdown、`localStorage`、設定ファイルへの保存を許可しない。
- API Key、Authorizationヘッダー、プロンプト本文、ノート本文をログ・エラー・診断情報へ出さない境界を決める。
- 接続先URLの許可範囲、HTTPS、ローカルプロバイダー、proxy、redirect、timeout、retry、レート制限、費用上限の扱いを決める。

### 2. AI設定・プロバイダー契約

- Go側のプロバイダー共通インターフェース、リクエスト・レスポンス・構造化エラーを決める。
- ストリーミング、キャンセル、再試行、部分応答、コンテキスト長、入力・出力上限を決める。
- プロバイダーごとの差分（認証、モデル名、endpoint、JSON形式、ストリーム形式）をどの層で吸収するか決める。
- 実キーを使わないテストdoubleと、接続確認の受け入れ条件を決める。

### 3. 生成結果の保存境界

- 要約、生成タイトル、タグ、分類、関連候補、Q&A、執筆結果を保存するか、保存する場合の正本（Markdown、SQLite、別成果物）を決める。
- 保存する生成結果の版、生成元モデル、生成日時、再生成・破棄方法、ユーザー編集との関係を決める。
- 生成結果を既存ノート本文へ自動適用するか、下書き・提案として確認を必須にするかを決める。
- 保存処理は既存のrevision/CAS、操作journal、ノート単位laneを壊さない契約にする。

### 4. WebDAV同期との境界

- Phase 3の同期契約ではAI生成結果を同期対象外としている。この前提を維持するか、Phase 4で新しい同期対象・スキーマ・競合契約を追加するかを決める。
- 同期する場合は、生成結果の識別子、更新競合、端末間の再生成・重複、outbox/change setへの含め方を決める。
- 同期しない場合は、どの端末で再生成するか、ローカルに残る生成結果をどう説明・削除するかを決める。
- Phase 3のWebDAV契約を未承認のまま変更せず、決定事項を別設計として記録する。

### 5. UI・データフロー

- 未設定、接続確認中、生成中、キャンセル、部分応答、失敗、再試行、利用制限、オフラインの状態を定義する。
- ノート本文を外部プロバイダーへ送る範囲、ユーザーへの確認表示、送信前のマスキング方針を決める。
- 空結果、大量ノート、長文、機密ノート、プロバイダー切り替え時の挙動を決める。

## 対象外

- Phase 4の設計承認前のAI API実装、DB migration、WebDAV契約変更
- Atlas Note側でのモデル学習、プロバイダーアカウントの作成・課金契約・請求管理
- ユーザーの明示確認なしのノート本文の上書き、秘密情報の自動同期
- 添付ファイル・履歴のAI連携（別スコープで扱う）

## Phase 4開始条件

- `todo-phese4.md` の必須設計・セキュリティ・受け入れ項目を完了する。
- AI認証、保存、同期境界、プロバイダー共通契約、UI状態、テスト方針をレビューで承認する。
- 実キーを使わない自動テスト、秘密情報非露出の確認、失敗時にローカル機能を継続できることの受け入れ条件を確定する。
