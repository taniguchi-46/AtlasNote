## todoの内容

Atlas Note に実装計画策定専用の `implementation-plan` Skill を導入する。

`.agents/skills/implementation-plan/SKILL.md` を作成し、機能実装、バグ修正、リファクタリングの前に既存コード、仕様、テスト、影響範囲を調査して、実装担当者がそのまま作業を開始できる具体的な実装計画を出力できるようにする。

この Skill は調査と計画策定のみを担当し、実装は行わない。

### 現状/前提

- `.agents/AGENTS.md` に Codex 固有の作業指針と参照順が定義されている。
- `docs/rules/ai.md`、`docs/rules/architecture.md`、`docs/rules/conventions.md`、`docs/rules/loop.md` に共通ルールと作業フローが定義されている。
- `docs/rules/loop.md` の作業フローは `Analyze → Search → Plan → Implement → Test → Review → Complete` であり、今回の Skill は `Analyze → Search → Plan` を専門化する。
- `.agents/skills/skill.md` に汎用開発 Skill が存在するが、実装まで含むため実装計画専用 Skill と責務が異なる。
- 既存ドキュメントと実コードに差異がある場合があるため、計画はドキュメントだけで確定せず実コードを確認して策定する必要がある。

### 実行手順

1. `.agents/AGENTS.md` と `docs/rules/` 配下の現行ルールを再確認する。
2. `.agents/skills/implementation-plan/` ディレクトリを作成する。
3. `.agents/skills/implementation-plan/SKILL.md` を作成する。
4. frontmatter に `name: implementation-plan` と、使用場面が明確な `description` を定義する。
5. Skill の入力として、機能実装、不具合修正、リファクタリング対象、TODOタスク、既存設計案を扱えるようにする。
6. 以下の調査フローを Skill に定義する。
   - 依頼内容、目的、完了条件、制約、対象範囲、対象外を整理する。
   - `README.md`、`docs/status.md`、`docs/rules/` 配下、関連開発ドキュメントを確認する。
   - 関連コード、呼び出し元、呼び出し先、状態管理、永続化、エラー処理、非同期処理、関連テスト、類似実装を調査する。
   - バグ修正では症状と根本原因を区別する。
   - 機能実装では現在の状態と要求との差分を特定する。
   - 直接変更する対象と影響確認だけ必要な対象を区別する。
   - 既存機能、データ保存、SQLite、Markdown Storage、Wails API、フロントエンド状態、非同期処理、終了処理、エラー処理、ユーザーデータへの影響を必要な範囲で確認する。
   - 保証する振る舞いを基準にテスト戦略を策定する。
7. 出力形式として以下を必須化する。
   - 結論
   - 調査結果
   - 現在の実装
   - 問題または不足
   - 関連ファイル
   - 実装方針
   - 順序付きの実装手順
   - テスト戦略
   - 確認コマンド
   - 影響範囲
   - リスクと注意点
   - 対象外
   - 未確定事項
8. 各実装手順に、変更対象、変更内容、変更理由、関連する既存処理、テストまたは確認方法を含める。
9. Skill の制約として、ソースコード、テスト、設定、ドキュメントの編集、ファイル作成・削除・リネーム、Git commit、Git push、PR作成を禁止し、計画策定後に停止することを明記する。
10. `.agents/AGENTS.md` の Skill 参照記述を確認し、必要な場合のみ `implementation-plan` の参照方法を追加または整理する。
11. 作成後に Skill をレビューし、既存ルールとの競合、古いパス参照、抽象的すぎる手順、実装開始を許す曖昧な表現がないことを確認する。

### 注意事項

- Skill の責務は「実装計画策定」の1つに限定する。
- 推測だけで計画を作成せず、実コードと既存ドキュメントを根拠にする。
- ドキュメントとコードが一致しない場合は、不一致を明示して実コードから現在の状態を確認する。
- 依頼内容にない仕様を追加しない。
- 無関係なリファクタリングを実装計画に含めない。
- リファクタリングが必要な場合は、要求を安全に実装するために必要な理由を明示する。
- テスト戦略を必須とし、バグ修正では可能な限り問題を再現する回帰テストを計画する。
- テストは内部実装の呼び出し回数ではなく、保証する振る舞いを基準に設計する。
- 確認コマンドはドキュメント上の想定だけで決めず、実際の設定ファイルや package scripts を確認する。
- `docs/architecture.md` や `docs/conventions.md` など、現在の配置と一致しない古いパスを新しい Skill に持ち込まない。

### 絶対遵守事項

- `implementation-plan` Skill の実行中は実装を開始しない。
- Skill は調査と計画策定で必ず停止する。
- ユーザーが実装も同時に依頼している場合でも、Skill 実行中はコードを変更しない。
- 実装は別作業または実装用 Skill に引き継ぐ。
- データ消失、上書き、競合、未保存状態に関係する変更は高リスクとして明示する。
- 実装担当者が追加調査なしで主要な変更を開始できる粒度まで具体化する。
